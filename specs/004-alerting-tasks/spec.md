# Feature Specification: Alerting Tasks Recipient

**Feature Branch**: `004-alerting-tasks`
**Created**: 2025-11-12
**Status**: Draft
**Input**: User description: "I want to build tasks inside Beacon alerting section. It would work in a way that I can choose Tasks as recipient on any subscription and then it would create a task record in database and I as a user could look at all the notifications sent to it, resolve it, track progress (how many records when query executes...)"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Create Task Recipient and View Generated Tasks (Priority: P1)

A user wants to create internal task tracking for high-priority query alerts (like critical data quality issues or compliance violations) without sending external notifications. They create a Tasks recipient, attach it to subscriptions, and view all generated tasks in a centralized task list.

**Why this priority**: This is the minimum viable product - users can create tasks and view them. It delivers immediate value by providing an internal notification channel that doesn't require external system integration.

**Independent Test**: Can be fully tested by creating a Tasks recipient, attaching it to a subscription, triggering query execution, and viewing the generated task in the task list. Delivers value as a self-contained notification archive.

**Acceptance Scenarios**:

1. **Given** I am on the Recipients page, **When** I click "Add Recipient" and select "Tasks" as the notification type, **Then** I can create a Tasks recipient with a name and description (no destination required)
2. **Given** I have a Tasks recipient created, **When** I attach it to a subscription and the scheduled query executes with results, **Then** a new task is created in the database
3. **Given** tasks have been generated from subscription executions, **When** I navigate to the Tasks page, **Then** I see a list of all tasks with subscription name, query name, execution date, result count, and resolution status
4. **Given** I am viewing the task list, **When** I filter by unresolved tasks, **Then** I see only tasks that have not been marked as resolved
5. **Given** I am viewing the task list, **When** I click on a specific task, **Then** I see task details including subscription info, query results, execution history, and resolution status

---

### User Story 2 - Resolve and Track Task Progress (Priority: P2)

A user reviews tasks generated from subscription executions and marks issues as resolved once they've been addressed. They can add notes documenting the resolution and track which tasks still need attention.

**Why this priority**: Adds workflow management capabilities. Users can now act on tasks and track completion, making this a functional task management system rather than just a notification log.

**Independent Test**: Can be tested by creating tasks (from P1), then marking them as resolved with notes. Validates that task state transitions work and resolution tracking is persistent.

**Acceptance Scenarios**:

1. **Given** I am viewing task details, **When** I click "Resolve Task" and optionally add resolution notes, **Then** the task is marked as resolved with a resolution timestamp
2. **Given** I have resolved a task, **When** I view the task list, **Then** the resolved task shows a "Resolved" badge and resolution date
3. **Given** I am viewing a resolved task, **When** I click "Reopen Task", **Then** the task returns to unresolved status
4. **Given** I have multiple tasks for the same subscription, **When** I view the task list, **Then** I can see the progression of result counts over time for that subscription

---

### User Story 3 - Track Subscription Progress with Task Timeline (Priority: P3)

A user wants to track how a subscription's query results change over time by viewing a timeline of all tasks generated for that subscription. This helps identify trends (e.g., "data quality issues increasing") or verify that a problem has been fixed (e.g., "no new alerts in 2 weeks").

**Why this priority**: Enhances analytical capabilities. Users can now understand patterns and trends rather than just managing individual tasks. This is valuable but not essential for basic task functionality.

**Independent Test**: Can be tested by generating multiple tasks over time for the same subscription, then viewing the timeline to verify chronological ordering and trend visualization.

**Acceptance Scenarios**:

1. **Given** I am viewing a subscription's details, **When** I navigate to the "Task History" tab, **Then** I see a chronological timeline of all tasks generated by that subscription
2. **Given** I am viewing the task timeline, **When** I examine each task entry, **Then** I see the execution date, result count, and resolution status
3. **Given** I am viewing the task timeline, **When** results show an increasing or decreasing trend, **Then** I can visually identify the pattern (e.g., chart or color coding)
4. **Given** a subscription has generated no new tasks in a specified period, **When** I view the timeline, **Then** I see a visual indicator of "No alerts" for that period

---

### User Story 4 - Manage Task Assignments and Notifications (Priority: P4)

A user wants to assign tasks to specific team members and receive notifications when new tasks are created for subscriptions they're monitoring. Team members can see tasks assigned to them and receive email/Teams notifications about new assignments.

**Why this priority**: Adds collaboration features. This is valuable for team environments but not essential for individual users or small teams who can manage tasks through the shared list.

**Independent Test**: Can be tested by assigning a task to a user and verifying they receive a notification and see it in their "Assigned to Me" view.

**Acceptance Scenarios**:

1. **Given** I am viewing a task, **When** I assign it to a team member, **Then** that person receives a notification about the assignment
2. **Given** tasks are assigned to me, **When** I navigate to the Tasks page, **Then** I can filter to see only "Assigned to Me" tasks
3. **Given** I am monitoring a subscription, **When** a new task is generated, **Then** I receive an email or Teams notification (based on my preferences)

---

### Edge Cases

- What happens when a subscription with a Tasks recipient executes but returns zero results? (Task should not be created, similar to NoResults notification status)
- What happens when a task is generated for a subscription that is later archived or deleted? (Task should remain accessible as historical record)
- What happens when query results are identical to the previous execution? (Task should not be created due to NotificationSilenced status)
- What happens when a query execution times out? (Task should not be created, timeout status should be recorded in QueryExecutionHistory only)
- What happens when a user tries to resolve an already-resolved task? (Should allow re-resolution with updated notes and timestamp, or prevent with clear message)
- What happens when query results exceed 10,000 records? (Task should store result count and summary, full results accessible via stored results if enabled)
- What happens when multiple subscriptions use the same Tasks recipient? (All tasks should be grouped/filterable by recipient in the UI)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST add "Tasks" as a new option in the NotificationType enum with value 4
- **FR-002**: System MUST allow users to create Tasks recipients with name, description, and NotificationType set to Tasks (no destination field required)
- **FR-003**: System MUST create a new Task entity to store task records with fields for subscription, query execution history, result count, created timestamp, resolved status, resolved timestamp, and resolution notes
- **FR-004**: System MUST create a TasksAdapter implementing the IAdapter interface to handle task creation when notifications are sent
- **FR-005**: System MUST create a task record only when NotificationStatus is NotificationSent (not for NoResults, Timeout, or NotificationSilenced statuses)
- **FR-006**: System MUST link tasks to QueryExecutionHistory records to track which execution generated the task
- **FR-007**: System MUST link tasks to Notification records (via NotificationId) to maintain consistency with existing notification architecture
- **FR-008**: System MUST provide a Tasks page in the UI showing a filterable, sortable list of all tasks with subscription name, query name, result count, created date, and resolution status
- **FR-009**: System MUST provide task details view showing full subscription information, query results (if stored), execution history, and resolution details
- **FR-010**: System MUST allow users to resolve tasks with optional resolution notes
- **FR-011**: System MUST allow users to reopen resolved tasks
- **FR-012**: System MUST allow users to filter tasks by resolution status (all, unresolved, resolved)
- **FR-013**: System MUST allow users to filter tasks by subscription or recipient
- **FR-014**: System MUST display task count trend data when viewing subscription details (number of tasks generated over time)
- **FR-015**: System MUST prevent creation of Tasks recipients with duplicate names within the same tenant
- **FR-016**: System MUST support archiving Tasks recipients (soft delete) following the existing ArchivableBaseEntity pattern
- **FR-017**: System MUST display task result count (number of query result records) in the task list and details
- **FR-018**: System MUST allow users to view stored query results from the task details page when subscription has StoreResults enabled
- **FR-019**: System MUST maintain task records even if the associated subscription or recipient is archived
- **FR-020**: System MUST provide task statistics on the Notifications/Tasks page (total tasks, unresolved count, resolved count, average resolution time)

### Key Entities

- **Task**: Represents an internal task record generated when a subscription with a Tasks recipient executes. Contains subscription reference, query execution reference, notification reference, result count, created timestamp, resolved status (boolean), resolved timestamp (nullable), resolved by user reference (nullable), resolution notes (nullable), and relationship to QueryExecutionHistory and Notification entities.

- **Tasks Recipient**: A recipient entity with NotificationType set to Tasks. Unlike Email/Teams/Jira recipients, does not require a destination field. Used as a target for subscriptions to generate internal task records rather than external notifications.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can create a Tasks recipient and attach it to a subscription in under 1 minute
- **SC-002**: Tasks are automatically generated within 5 seconds of query execution completion
- **SC-003**: Users can view task list with 1000+ tasks without performance degradation (load time under 2 seconds)
- **SC-004**: Users can filter and find specific tasks in under 10 seconds
- **SC-005**: Task resolution (marking as resolved) completes in under 1 second
- **SC-006**: 90% of users successfully create their first Tasks recipient without assistance
- **SC-007**: Task list page supports sorting and filtering without page reload (client-side operations)
- **SC-008**: Users can identify unresolved tasks requiring attention within 5 seconds of opening the Tasks page
- **SC-009**: System maintains task history integrity even when subscriptions are archived (100% retention of historical task data)

## Assumptions *(if applicable)*

- Tasks will be tenant-scoped (tasks are only visible within the tenant that created them)
- Tasks do not require external system integration or API endpoints (internal-only feature)
- Task resolution is a manual user action (no automatic resolution based on subsequent query results)
- Tasks inherit the existing security model where users can only view tasks for subscriptions they have access to
- The existing Notification entity will be created for Tasks recipients (following the same pattern as Email/Teams/Jira)
- Task assignment and notification features (P4) may be deferred to a future phase if time constraints exist
- Users access tasks through a dedicated "Tasks" navigation item in the alerting section (similar to Recipients, Subscriptions, Notifications)
- Task filtering and sorting will use client-side operations for better performance (similar to existing Beacon grid components)
- Resolution notes are optional (users can resolve tasks without providing notes)
- Only one resolution record per task (no resolution history tracking in initial version)

## Out of Scope *(if applicable)*

- Task prioritization levels (high/medium/low) - all tasks have equal priority in initial version
- Task due dates or SLA tracking
- Task comments or discussion threads (only single resolution note supported)
- Task dependencies (linking related tasks)
- Bulk task operations (resolve multiple tasks at once)
- Task templates or custom task fields
- Email/Teams notifications when new tasks are created (part of P4, may be future phase)
- Task assignment to specific users (part of P4, may be future phase)
- Integration with external task management systems (Jira already exists as separate recipient type)
- Task export functionality (users can export query results, but not task metadata)
- Task recurring patterns or scheduled followups

## Dependencies *(if applicable)*

- Existing NotificationType enum (must add new "Tasks" value)
- Existing IAdapter interface and AdapterFactory (must register TasksAdapter)
- Existing Recipient entity (reused for Tasks recipients)
- Existing Notification entity (tasks must link to notifications for consistency)
- Existing QueryExecutionHistory entity (tasks must link to execution history)
- Existing notification sending flow in JobService (must invoke TasksAdapter)
- Existing RecipientService for Tasks recipient CRUD operations
- Existing Beacon UI grid components for task list rendering
- Existing authentication/authorization system (tasks respect user permissions)

## Technical Constraints *(if applicable)*

- Tasks must follow the existing adapter pattern (implement IAdapter interface)
- Tasks recipient must use NotificationType enum (cannot introduce new recipient type system)
- Task entity must follow naming conventions (PascalCase, inherit from appropriate base entity)
- Task creation must be synchronous within the notification sending flow (no separate background job)
- Task UI must use Blazor components consistent with existing Beacon.UI patterns
- Database schema changes require migration (user creates migrations manually per CLAUDE.md guidance)
- Tasks must respect the existing soft delete pattern (if Task entity is archivable)
- Task list must support the existing search/filter patterns used in Recipients and Subscriptions pages
