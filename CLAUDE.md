## 1. Security & Compliance

§1.1 **Connection string encryption**: All data source connection strings are encrypted at rest using `Semantico:EncryptionKey` (required config). Never store plaintext connection strings.

§1.2 **API key hashing**: API keys are hashed with SHA256 before storage. Raw keys are shown to the user only once at creation.

§1.3 **MCP query guardrails**: All MCP-executed queries are read-only enforced. PII detection/masking and row limits are configurable per project.

§1.4 **Cookie security**: `Semantico.Auth` cookie uses `HttpOnly = true`, `SameSite = Lax`, `SecurePolicy = SameAsRequest`.

§1.5 **No secrets in code**: Never hardcode connection strings, API keys, or encryption keys. All sensitive values come from `appsettings.json` or environment variables.

§1.6 **SQL parameterization**: When using Dapper or raw SQL, always use parameterized queries. Never interpolate user input into SQL strings.

## 2. Architecture Patterns

§2.1 **CQRS via MediatR**: All UI actions go through `IMediator.Send()` with query/command records and handlers in `Semantico.Core/Handlers/` and `Semantico.AI/Handlers/`. Each project registers its own assembly: `services.AddMediatR(cfg => cfg.RegisterServicesFromAssembly(...))`.

§2.2 **Handler file convention**: One handler per file. The handler class, request record, and result record all live in the same file. Handler class is `internal sealed class` using primary constructors.

```csharp
// Good — see Semantico.Core/Handlers/QueryFolders/CreateQueryFolderHandler.cs
internal sealed class CreateQueryFolderHandler(IDbContextFactory<SemanticoContext> contextFactory)
    : IRequestHandler<CreateQueryFolderCommand, CreateQueryFolderResult>
{
    public async Task<CreateQueryFolderResult> Handle(...)
    { ... }
}

public record CreateQueryFolderCommand(...) : IRequest<CreateQueryFolderResult>;
public record CreateQueryFolderResult(...);
```

§2.3 **Request naming**: Commands use `{Action}{Domain}Command`, queries use `{Action}{Domain}Query`. Results use `{Action}{Domain}Result`.

§2.4 **No repository pattern**: Handlers and services access `SemanticoContext` directly via `IDbContextFactory<SemanticoContext>`. Do not introduce repository abstractions.

§2.5 **No domain events**: Cross-concern communication happens through direct service calls, not MediatR notifications.

§2.6 **No AutoMapper/mapping libraries**: All mapping is manual — use `.Select()` projections in LINQ or inline property assignment.

§2.7 **Builder pattern for DI**: `SemanticoBuilder` enables fluent registration chaining. Connectors register via `Add{Engine}Connector()` extension methods on `SemanticoBuilder`.

§2.8 **Connector/Provider pattern**: Data source connectors implement `IDataSourceProvider` and register with `ConnectorRegistry`. See `Semantico.Connector.PostgreSql/ServiceCollectionExtensions.cs` for the canonical example.

§2.9 **DI registration per project**: Each project has `ServiceConfiguration.cs` (Core, AI, MCP) or `ServiceCollectionExtensions.cs` (Connectors, database providers). Never register services inline in `Program.cs`.

§2.10 **DI lifetimes**: Singletons for configuration and provider managers. Scoped for handlers (MediatR default) and user context. Transient for service classes (`TryAddTransient` is the standard in AI project). Use `TryAdd*` to allow host-level overrides.

§2.11 **Soft delete via global query filters**: Entities inheriting `ArchivableBaseEntity` get automatic `HasQueryFilter(x => x.ArchivedTime == null)`. Call `.Archive()` instead of deleting. Use `.IgnoreQueryFilters()` only when you explicitly need archived records.

§2.12 **Background jobs via Hangfire**: All recurring/scheduled work uses Hangfire with PostgreSQL storage. Do not use `BackgroundService` or `IHostedService`. Register recurring jobs in `Program.cs`.

## 3. Coding Style

*Based on `.claude/references/coding-guidelines.md` — adapted to this project's conventions.*

### Naming

§3.1 **PascalCase** for class names, method names, public properties, and events.

§3.2 **camelCase** for local variables and method arguments.

§3.3 **_camelCase** for private fields (prefixed with underscore). Note: most Core handlers use primary constructors, so private fields are less common there.

§3.4 **I-prefix** for interfaces: `IKnowledgeGraphService`, `IDataSourceProvider`.

§3.5 **Avoid abbreviations** except standard ones (Id, Xml, Uri). Acronyms: only first letter uppercase (`personOib`, not `personOIB`).

### Lambda & LINQ

§3.6 **Lambda parameter `x`**: Use `x` for lambda parameters. Inner lambdas use `y`, then `z`.

```csharp
// Good
var activeProjects = projects
    .Where(x => x.ArchivedTime == null)
    .Where(x => x.DataSources
        .Any(y => y.IsEnabled))
    .ToList();
```

§3.7 **Chain methods on separate lines**: Each `.Where()`, `.Select()`, `.OrderBy()`, etc. on its own line.

§3.8 **Separate `&&` into multiple `.Where()` calls**:

```csharp
// Bad
.Where(x => x.Status == Active && x.CreatedTime > cutoff)

// Good
.Where(x => x.Status == Active)
.Where(x => x.CreatedTime > cutoff)
```

§3.9 **Use `.Where()` before `.FirstOrDefault()` / `.SingleOrDefault()`**:

```csharp
// Bad
var folder = context.QueryFolders.FirstOrDefault(x => x.Id == id);

// Good
var folder = await context.QueryFolders
    .Where(x => x.Id == id)
    .FirstOrDefaultAsync(cancellationToken);
```

§3.10 **`.Select()` `new` on its own line**: Place `new` keyword and class name on a new line indented one level deeper than `.Select()`:

```csharp
// Good — see Semantico.Core/Handlers/Projects/GetProjectsHandler.cs
var projects = await context.Projects
    .Where(x => x.ArchivedTime == null)
    .OrderByDescending(x => x.CreatedTime)
    .Select(x =>
        new ProjectListItem
        {
            Id = x.Id,
            Name = x.Name,
            DataSourceCount = x.DataSources.Count
        })
    .ToListAsync(cancellationToken);
```

§3.11 **Prefer async LINQ for database queries**: Always use `ToListAsync()`, `FirstOrDefaultAsync()`, `CountAsync()`, etc. with `CancellationToken`.

§3.12 **Use `foreach` over LINQ `.ForEach()`**: Never call `.ToList().ForEach(x => ...)`.

§3.13 **From database, only select data which will be used**: Use `.Select()` projections instead of `.Include()`. No need for `.Include()` when using `.Select(new ...)`.

§3.14 **Do not use `Single` when fetching by primary key**: Use `FirstOrDefault` (entities either exist or don't; duplicates are impossible for PKs).

### Layout

§3.15 **File-scoped namespaces**: Always use `namespace Semantico.Core.Handlers.Queries;` (no braces).

§3.16 **One statement per line, one declaration per line**.

§3.17 **Match namespaces to folder structure**.

§3.18 **`using` directives outside namespace**: Sort `System.*` first, then others alphabetically.

§3.19 **Simple `using` declarations**: `using var context = ...` not `using (var context = ...) { }`.

§3.20 **Blank line before `return`** in multi-line methods.

§3.21 **Never use 2 empty lines in a row**.

§3.22 **Private methods last in file**.

§3.23 **Private fields grouped without blank lines between them**, then a blank line before the first method/constructor.

§3.24 **Separate logically-related code blocks** with a single blank line. Separate unrelated `if` blocks with a blank line.

§3.25 **Avoid `else` — return/throw early**:

```csharp
// Good
if (folder == null)
{
    throw new InvalidOperationException("Folder not found.");
}

// Happy path continues here
```

§3.26 **Always use `var`** for local variables (implicit typing).

§3.27 **Always use braces** for `if`, `else`, `while`, `for`, `foreach` — even for single-line bodies.

§3.28 **Object initializer without `()`**: `new Person { Name = "Name" }` not `new Person() { Name = "Name" }`.

§3.29 **Assign all properties in object initializer** — don't set properties after construction.

§3.30 **Extract complex conditions to named variables**.

§3.31 **Prefer null coalescing** (`??`) over ternary null checks.

§3.32 **Prefer ternary for simple assignments** over `if-else`.

§3.33 **No `this.` prefix** on field access.

§3.34 **No meaningless comments**: Don't add `/// <param name="x">the x parameter</param>`.

§3.35 **Add `.Add()` / `.AddAsync()` calls just before `SaveChangesAsync()`** — not earlier in the method.

§3.36 **Avoid helper lists**: Use `.Select()` or methods with `yield return` instead of manually building lists.

§3.37 **Declare variables close to first use** — don't declare at method top.

§3.38 **Avoid initializing collections in loop headers**: Materialize to a variable first, then iterate.

## 4. Testing Standards

§4.1 **Framework**: NUnit 4 + Moq + FluentAssertions + bUnit. All in `Semantico.Tests/`.

§4.2 **Test naming**: `{QueryOrFeatureName}_Translates()` for query translation tests. Use descriptive names for other tests.

§4.3 **Query translation testing**: The primary test strategy validates EF Core LINQ → PostgreSQL SQL translation via `ToQueryString()` without hitting a real database. See `Semantico.Tests/Integration/QueryTranslationTests.cs`.

§4.4 **Test context**: Use `NpgsqlTestContext.Create()` for translation tests — creates a context with the Npgsql provider and snake_case naming convention using a dummy connection string.

§4.5 **When adding handlers**: Add a corresponding query translation test in `QueryTranslationTests.cs` if the handler contains non-trivial LINQ queries.

§4.6 **No InMemory provider**: Do not use `UseInMemoryDatabase` for tests — it doesn't catch provider-specific translation issues.

## 5. Data Layer

§5.1 **DbContext factory pattern**: Inject `IDbContextFactory<SemanticoContext>`, create short-lived contexts via `await contextFactory.CreateDbContextAsync(cancellationToken)`. Never inject `SemanticoContext` directly.

§5.2 **Abstract context with provider subclasses**: `SemanticoContext` is abstract. `PostgreSqlSemanticoContext` and `SqlServerSemanticoContext` provide concrete implementations. PostgreSQL uses `UseSnakeCaseNamingConvention()`.

§5.3 **Configurable schema**: Default schema is `"semantico"`. All entities live in this schema.

§5.4 **Fluent API configuration only**: All entity configuration in `SemanticoContext.OnModelCreating()` via private `Configure{Domain}Entities()` methods. Do not use data annotations (`[Table]`, `[Key]`, etc.) or separate `IEntityTypeConfiguration<T>` classes.

§5.5 **Entity base classes**:
- `BaseEntity`: `Id` (int), `CreatedTime` (DateTime, defaults to `DateTime.UtcNow`)
- `ArchivableBaseEntity : BaseEntity`: adds `ArchivedTime` for soft delete
- `AuditableBaseEntity : BaseEntity`: adds `CreatedByUserId`, `ModifiedTime`, etc.

§5.6 **Select projections over Include**: Use `.Select(x => new { ... })` for read queries. `.Include()` is only needed when you need navigation properties for writes or complex operations (82 usages vs 420 `.Select()` usages).

§5.7 **One `SaveChangesAsync()` per handler**: Services called by handlers must not call `SaveChanges`. The handler owns the unit of work.

§5.8 **Dual migrations**: Both PostgreSQL (`Semantico.Core.PostgreSql`) and SQL Server (`Semantico.Core.SqlServer`) need migrations when entities change. See memory file `migrations-workflow.md` for the process.

§5.9 **Dapper for specialized queries**: Dapper is used alongside EF Core for metadata extraction (`*MetadataExtractor.cs`), bulk operations, and performance-critical raw SQL. Always parameterize.

§5.10 **Bulk operations**: Use `EFCore.BulkExtensions` (`BulkInsertAsync`, `BulkUpdateAsync`) via `Semantico.Core/Helpers/BulkHelpers.cs` for large datasets.

## 6. Performance Standards

§6.1 **LLM rate limiting**: All LLM calls go through `LlmRequestQueue` with configurable `MaxConcurrentRequests`. Never call LLM providers directly.

§6.2 **Memory cache for hot paths**: `IMemoryCache` is used for MCP settings, metadata, and app settings. Use cache for data that's read frequently but changes rarely.

§6.3 **Kestrel timeouts**: Keep-alive and request header timeouts are set to 5 minutes for AI operations. HttpClient default timeout is also 5 minutes.

§6.4 **Hangfire polling**: Queue poll interval is 1 second. Automatic retry is disabled (`Attempts = 0`) — failures are investigated, not retried blindly.

§6.5 **In-memory SQLite for cross-source joins**: Multi-step queries cache intermediate results in `Microsoft.Data.Sqlite` in-memory database, then run a final join step.

## 9. Project-Specific Patterns

§9.1 **Error handling**: Throw `InvalidOperationException` for business rule violations (entity not found, duplicate name, invalid state). Throw custom `SemanticoException` for domain-specific errors. No Result pattern or FluentValidation.

§9.2 **Authentication stack**: Cookie auth for Blazor UI, API key auth for MCP, optional JWT bearer. Middleware chain: `ApiKeyAuthMiddleware` → `UseAuthentication` → `UseAuthorization` → `SemanticoCookieAuthMiddleware` → `LoginFormAuthMiddleware`.

§9.3 **MCP tool architecture**: Tools are classes with `[McpServerTool]` attributes. They resolve project context from the MCP session. `McpSignalService` records usage signals for the learning system.

§9.4 **LLM provider swapping**: `LlmProviderManager` holds the active provider (OpenAI, Azure OpenAI, Claude, Bedrock). `DelegatingLlmProvider` proxies calls to the current active provider. Provider can be changed at runtime via admin settings.

§9.5 **MudBlazor UI pitfalls**:
- `MudSelectItem` with `Value="@null"` causes dropdown malfunction — use `Placeholder` + `Clearable` instead.
- `MudForm @bind-IsValid` may not track reliably — always call `await _form.Validate()` before checking `_isValid`. Add `Immediate="true"` on required fields.

§9.6 **EF Core + PostgreSQL pitfalls**: Empty sequence aggregates (e.g., `Max()` on empty set) and Npgsql translation issues. See memory for details.

§9.7 **No `.Include()` needed with `.Select()`**: When projecting with `.Select(new ...)`, EF Core automatically joins related tables. Adding `.Include()` is unnecessary and can hurt performance.

## Key Features

### AI Actors (Autonomous Monitoring Agents)
- AI-powered agents that autonomously monitor data sources and generate insights
- Plan-approve-reject workflow: LLM generates analysis plan → human reviews → approved plans create queries/subscriptions
- Handlers: `Semantico.Core/Handlers/Ai/AiActor/`, `Semantico.AI/Handlers/Ai/AiActor/`
- UI: `/ai-actors`, `/ai-actors/{ActorId:int}`, `/ai-actors/data-source/{DataSourceId:int}`

### Queries & Query Management
- CRUD for SQL queries with parameter binding, multi-step cross-source joins
- Hierarchical folder organization, version control, lock/unlock, change history
- Handlers: `Semantico.Core/Handlers/Queries/`, `Semantico.Core/Handlers/QueryVersions/`, `Semantico.Core/Handlers/QueryFolders/`
- UI: `/queries`, `/queries/{id:int}`, `/queries/add`

### MCP Server (Project-Centric)
- Streamable HTTP at `/semantico/mcp`
- **5 tools**: `get_context`, `ask` (NL-to-SQL), `query` (direct SQL), `get_documentation`, `search`
- **4 resources per project**: documentation, schema, quality report, comprehensive report
- `ask` tool: two-phase LLM routing → SQL generation. Cross-source queries use in-memory SQLite joins.
- Query guardrails: read-only enforcement, PII detection/masking, row limits
- Audit logging via `McpAuditService`, learning signals via `McpSignalService`
- Handlers: `Semantico.Core/Handlers/McpSettings/`

### Data Sources
- Multi-connector: PostgreSQL, SQL Server, MySQL, BigQuery, Snowflake, Databricks, Azure Synapse, CloudWatch, REST API
- All connectors implement `IDataSourceProvider`, registered via builder pattern
- UI: `/data-sources`, `/data-sources/{DataSourceId:int}`

### Projects & Knowledge Graph
- Multi-source projects linking data sources and GitHub repositories
- AI-generated documentation (multi-pass LLM), exportable as Markdown/HTML
- Code-to-data lineage via `CodeReference` entities
- UI: `/semantico/projects`, `/semantico/projects/{ProjectId:int}`

### Dashboards, Data Quality, Subscriptions, Notifications
- Custom dashboards with KPI/Chart/Gauge/Table widgets and role-based permissions
- Data contracts with rules, weighted scoring, scheduled evaluations
- Cron-scheduled query execution with recipient routing
- Alert delivery via Teams, Slack, Email, Webhook

### API Key Management
- Scoped API keys (Read, Execute, Admin) with project restrictions
- UI: `/semantico/settings/api-keys` (admin only)

### AI Services
- LLM Providers: OpenAI, Azure OpenAI, Claude (Anthropic), AWS Bedrock — swappable at runtime
- Semantic Search, Knowledge Graph, dbt Integration, Alert Generation
- Rate-limited concurrent LLM requests via `LlmRequestQueue`
